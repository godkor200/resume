## 👋 자기소개

안녕하세요. 5년차 백엔드 개발자 유병국입니다.

 저는 개발을 할 때 “이 기능이 돌아가느냐”보다 “이 구조가 오래 버틸 수 있느냐”를 더 중요하게 생각합니다. 처음에는 작은 기능처럼 보여도 서비스가 커지면 결국 데이터 구조와 운영 방식의 차이가 성능과 유지보수성 차이로 이어진다는 점을 여러 번 경험했기 때문입니다.

 실무에서는 집계 성능 문제나 동시성 문제를 겪으면서 운영 이슈를 단순 수정이 아니라 구조 개선 관점에서 풀어가는 기준을 갖게 됐습니다. 지금은 백엔드를 구현할 때도 데이터 정합성과 조회 성능 그리고 운영 안정성을 함께 보는 편입니다.

 저는 기획과 운영의 요구를 실제 서비스 구조로 연결할 수 있는 개발자이고 싶습니다. 예전 영업 경험 덕분에 현업의 언어와 개발의 언어 사이를 이해하고 조율하는 데 익숙해졌고 개발자로 일한 뒤에는 그 감각을 서비스 설계와 운영 관점으로 이어오고 있습니다.

 다른 사람이 보더라도 흐름을 이해할 수 있는 구조를 만드는 개발자가 되고 싶습니다.

    → 스크래핑, 적재, 가공, 집계 흐름을 분리해 운영 중 오류를 빠르게 확인하고 재처리 가능한 구조로 개선할 수 있습니다.

- **유지보수 가능한 코드베이스 정리**
    
    → 테스트, 문서화, 책임 분리를 통해 다른 개발자도 이해하고 이어서 개발할 수 있는 코드 구조를 만드는 데 기여할 수 있습니다.
    
- **반복 업무 생산성 향상**

### 2. 주요 성과

- 백엔드 아키텍처와 API 전반 설계 및 구현
- 데이터 수집, 가공, 집계 파이프라인 설계
- DB 설계와 인덱스 최적화
- 기존 Blip 서비스와 신규 K-POP Radar 사이의 데이터 동기화 구조와 조회 모델 재설계, nestjs 마이그레이션 참여
- 사전 집계 구조와 캐시 적용으로 조회 성능과 운영 안정성 개선
- 기사 썸네일 이미지 해시 비교 기반 중복 제거 로직 구현

### 3. 처음 마주한 문제

 기존 Blip 서비스의 운영 데이터는 MySQL에 계속 쌓이고 있었고,새로 구축하는 K-POP Radar는 PostgreSQL 기반으로 설계해야 했기 때문에 한동안 데이터베이스가 이원화된 상태였습니다.

 단순히 데이터를 옮기는 문제가 아니라 MySQL 데이터를 주기적으로 동기화하면서도 새 서비스 화면 구조와 기획 의도에 맞는 PostgreSQL 기준 데이터 모델을 다시 설계해야 했고 핵심은 기존 데이터를 그대로 이전하는 것이 아니라 서비스에서 실제로 사용하는 지표를 기준으로 조회 구조를 다시 정의하는 일이었습니다.

### 4. 고민했던 사항

- 모든 데이터를 실시간으로 조회할 것인가 아니면 일부를 사전 집계할 것인가
- 조회 API와 집계 로직 그리고 캐시 구조를 어디까지 분리할 것인가
- 이원화된 데이터 흐름 안에서 정합성과 운영 안정성을 어떻게 유지할 것인가
- 수집한 기사 데이터를 어떤 기준으로 중복 제거할 것인가

### 5. 최종 결정과 해결 방법

 기존 Blip 데이터는 MySQL에 계속 누적되고 있었고 새 서비스는 PostgreSQL을 기준으로 설계해야 했기 때문에 주기적 동기화 구조와 새 서비스용 데이터 모델 설계를 함께 진행했습니다.

화면에서 자주 호출되고 계산 비용이 큰 지표는 사전 집계 테이블로 분리했습니다. 반대로 비교나 탐색 성격이 강한 데이터만 동적 조회로 남겼습니다.

 여기에 실제 쿼리 패턴 기반 인덱스 최적화, 시계열 데이터 파티셔닝, Redis 캐시를 함께 적용해 조회 병목을 줄였습니다.

 기사 데이터는 썸네일 이미지 전처리 후 해시 기반 비교로 중복 콘텐츠를 식별하고 대표 기사만 남기도록 처리했습니다.

 운영 단계에서는 동기화 누락, 집계 지연, 이상치를 빠르게 확인할 수 있도록 모니터링과 Slack 알림 구조를 함께 두었습니다. 결국 단순 이전이 아니라 이원화된 데이터 흐름을 감당하면서 새 서비스와 기존 서비스에 맞는 조회 모델을 다시 설계한 것이 핵심이었습니다.

### 6. 구조 / 아키텍처

- Backend: NestJS, TypeORM
- Database: PostgreSQL, GCP Firestore, Redis
- Infra: Kubernetes, Argo CD

  db-syncer는 Firestore의 Raw 데이터를 PostgreSQL 기준 모델로 동기화하면서 정규화, 비정규화, 집계까지 함께 수행했습니다.

 서비스는 PostgreSQL 기준 데이터를 조회하고 사용자 세션과 빈번한 조회 데이터는 Redis로 분리했습니다.

### 7. 회고

초기에는 Raw 데이터와 가공 단계를 분리해 재처리 유연성을 확보하는 데 무게를 뒀습니다. 운영을 해보니 유연성만큼 중요한 건 비용과 관리 복잡도였습니다. 특히 데이터 단계가 많아질수록 집계 완료 여부, 이상치 감지, 수집 실패 추적 같은 운영 장치가 더 중요해졌고 이를 위해 Slack 기반 알림 로직도 함께 만들었습니다.

 이 경험을 통해 데이터 서비스는 많이 모으는 것보다 화면에 필요한 지표를 안정적으로 제공하는 구조가 더 중요하다는 점을 배웠습니다.

 다시 만든다면 재처리 가능성보다 운영 단순화를 더 우선해 PostgreSQL 중심의 더 단순한 구조를 먼저 가져갈 것 같습니다.

## 2️⃣ 두디스 (Dothis)

### 1. 프로젝트 설명

- 유튜브 콘텐츠 소재 분석 / 키워드 트렌드 분석 플랫폼

 유튜브 크롤링 데이터를 기반으로 키워드 분석과 소재 분석 그리고 대용량 집계 API를 제공하는 데이터 분석 플랫폼이었습니다.

### 2. 주요 성과

- 백엔드 API 설계 및 데이터 조회와 집계 성능 개선
- ETL, 적재, 인덱싱 구조 설계 협업 및 구현 지원
- 대용량 텍스트 데이터 처리 구조 설계
- 비용 제약 안에서 운영 가능한 아키텍처 방향 정리
- CI/CD 구축을 통한 배포 안정성 향상

### 3. 처음 마주한 문제

초기에는 MySQL에서 크롤링 데이터를 조인하고 집계해 API를 제공했습니다. 데이터가 쌓이면서 병목이 심해졌고 핵심 집계 API 하나를 호출하면 40초에서 1분까지 걸리는 상황이 발생했습니다. 문제는 단순히 느린 쿼리 하나가 아니었습니다. 조인 중심 구조 자체가 조회 목적과 맞지 않았고 시간이 갈수록 한계가 더 분명해졌습니다.

### 4. 고민했던 선택지

- RDB 중심 구조를 유지하면서 스케일업과 튜닝으로 버틸 것인가
- ETL을 도입해 검색과 집계 전용 스토리지를 분리할 것인가
- 인메모리 구조로 조인을 더 빠르게 할 것인가
- 비정규화로 조인 자체를 줄일 것인가

### 5. 최종 결정과 해결 방법

자세한 의사결정 과정은 아래 글에 정리했습니다.

 참고: [https://velog.io/@godkor200/왜-세-번이나-데이터베이스를-바꿨을까-빅데이터-플랫폼의-디비-설계-과정](https://velog.io/@godkor200/%EC%99%9C-%EC%84%B8-%EB%B2%88%EC%9D%B4%EB%82%98-%EB%8D%B0%EC%9D%B4%ED%84%B0%EB%B2%A0%EC%9D%B4%EC%8A%A4%EB%A5%BC-%EB%B0%94%EA%BF%A8%EC%9D%84%EA%B9%8C-%EB%B9%85%EB%8D%B0%EC%9D%B4%ED%84%B0-%ED%94%8C%EB%9E%AB%ED%8F%BC%EC%9D%98-%EB%94%94%EB%B9%84-%EC%84%A4%EA%B3%84-%EA%B3%BC%EC%A0%95)

처음에는 AWS 기술 자문을 계기로 OpenSearch 기반 구조를 도입해 검색과 집계 성격의 워크로드를 분리했습니다. MySQL에서 40초 이상 걸리던 집계 쿼리가 20초대까지 줄었지만 구조적으로 완전히 해결된 수준은 아니었습니다. 이후 비용과 운영 복잡도가 커졌고 Ignite 같은 인메모리 구조도 검토했지만 대용량 시계열 데이터와 운영 안정성 측면에서 적합하지 않았습니다.

 결국 문제의 핵심은 조인을 빠르게 하는 것이 아니라 조인을 계속 요구하는 구조 자체에 있다고 판단했습니다.

 그래서 DWH 데이터를 Logstash로 비정규화해 OpenSearch에 인덱싱하고 조회 시점 조인을 최대한 없애는 방식으로 전환했습니다. 검색 요청은 토큰 테이블에서 키워드와 연결된 콘텐츠 ID를 먼저 찾고 그 후보군을 기준으로 OpenSearch의 비정규화 데이터에서 최종 조회를 수행하도록 설계했습니다. 그 결과 핵심 집계 API 응답 속도를 약 40초에서 3초 수준까지 줄일 수 있었습니다.

### 6. 구조 / 아키텍처

- 메인 저장소 : mySQL
- 데이터 수집: Kafka로 수집 계층 분리
- 기준 저장소(DHW): MariaDB
- 인덱싱: Logstash → OpenSearch
- 검색 보조(Cache): 토큰 테이블, Redis
- 텍스트 처리: 임베딩, 토크나이징, 클러스터링

- DWH에서 크롤링된 영상들을 파티셔닝을 하기 위해 텍스트 임베딩과 토크나이징, 클러스터링을 거쳤습니다. 결과물을 바탕으로 검색 키워드와 토큰 테이블을 만들었습니다. 이 과정에서 BERT 기반 임베딩이나 토크나이징은 단순 문장 저장이 아니라 키워드와 문맥을 더 잘 반영한 검색 단위를 만들기 위한 시도였습니다.

## 3️⃣ 프린트시티 아이엠디자인 (Iam Design)

### 1. 프로젝트 설명

- 인쇄물 커머스 플랫폼 자동화 시스템

 명함, 전단지, 종이류, 포토카드 같은 인쇄물 상품을 다루는 커머스 프로젝트였습니다. 기존에는 .NET 기반으로 상품 상세 페이지를 모두 하드코딩으로 생성하고 관리하고 있었습니다. 이 프로젝트에서는 관리자 페이지에서 원재료, 옵션, 조합을 정의하면 해당 조합을 기반으로 상품 상세 페이지가 자동 생성되는 구조로 전환하는 것이 목표였습니다.

### 2. 주요 성과

- 상품 상세 페이지 자동 생성 로직 설계 및 백엔드 API 구현
- 신규 개발팀의 개발 컨벤션과 배포 프로세스 표준화
- 관리자 API 및 데이터 모델링
- 상품 옵션과 원재료 조합 구조 설계
- 재고 동시성 문제 분석 및 해결
- Express 기반 구조 개선
- NestJS + Hexagonal Architecture 전환
- CI/CD 구축 및 운영 자동화

### 3. 처음 마주한 문제

 기존 구조는 상품 하나당 상세 페이지 하나를 직접 하드코딩하는 방식이었습니다. 상품이 늘어날수록 페이지 수가 같이 늘었고 수정 범위도 커졌습니다. 운영 변경이 생길 때마다 개발이 같이 들어가야 했고 레거시 파일 복사 기반 유지보수 때문에 휴먼 에러도 자주 발생했습니다. 여기에 주문이 동시에 들어오면 재고 불일치 문제가 생겼고 원재료를 실시간 조인으로 화면에 구성하려 하면 렌더링 성능과 운영 복잡도가 함께 커지는 구조였습니다.

### 4. 고민했던 선택지

- 원재료를 조인으로 가져오는 방식과 document DB 기반 조합 관리 방식 중 무엇이 더 적합한가
- 원재료 재고의 실시간 정합성과 페이지 렌더링 속도 사이에서 어떤 균형을 잡을 것인가
- 복잡한 .NET 구조를 유지할 것인가 아니면 Express와 NestJS로 점진 전환할 것인가
- 배포 과정에서 반복되던 휴먼 에러를 어떻게 줄일 것인가

### 5. 최종 결정과 해결 방법

상품을 페이지 단위가 아니라 조합 데이터 단위로 다시 정의했습니다. 관리자 페이지에서 기존 원재료와 옵션을 조합해 MongoDB document로 관리하고 Next.js 공통 템플릿이 그 JSON 구조를 바탕으로 조건부 렌더링되도록 설계했습니다.

 이를 통해 상품 추가 시 개발 작업을 반복하지 않아도 됐고 운영자가 조합 데이터만 정의하면 동일한 템플릿 기반에서 페이지 구성이 달라지는 구조를 만들 수 있었습니다.

 재고 관리에는 Optimistic Locking을 적용했습니다. 버전 기반으로 충돌을 감지하고 재시도 가능한 흐름을 설계해 동시 주문 환경에서도 정합성을 유지하도록 했습니다. 기존 구조는 .NET에서 Express를 거쳐 최종적으로 NestJS와 Hexagonal Architecture로 정리했습니다.

 도메인 중심으로 역할과 책임을 나누고 기능 변경 시 영향 범위를 줄이는 방향으로 구조를 개선했습니다.

### 6. 구조 / 아키텍처

- Backend: NestJS
- Database: MongoDB
- Infra: AWS EC2, ECS
- CI/CD: GitHub Actions

### 7. 운영 중 발생한 문제와 개선

- 재고 동시성 문제 → Optimistic Locking으로 해결
- 코드 변경 영향 범위 불명확 → 도메인 분리와 계층 구조 도입
- 배포 안정성 부족 → CI/CD 파이프라인 구축
- 코드 상태 관리 혼선 → Git 기반 협업 구조 정리

### 8. 회고

 하드코딩으로 페이지를 하나씩 추가하는 방식은 문제를 해결하는 것이 아니라 같은 문제를 계속 복제하는 구조일 수 있다는 걸 이 프로젝트를 통해 체감했습니다.

 조합 데이터 기반 구조로 전환하면서 운영자가 직접 상품을 관리할 수 있게 했고 재고 관리에는 Optimistic Locking을 적용해 정합성을 확보했습니다.

 이 경험 이후로는 기능을 더하는 것보다 반복되는 변경과 동시성 상황에서도 버틸 수 있는 구조를 먼저 만드는 쪽으로 기준이 바뀌었습니다.
